01{% trans "N+1 Problem - ForeignKey" %} {% trans "PROBLEM" %}

{% trans "The N+1 problem is the most common in Django applications. It happens when you fetch a list of objects (1 query) and then access a relationship on each object inside a loop (N queries) - resulting in N+1 queries total." %}

{% trans "How it happens" %}

{% trans "Imagine you have 50 books and want to display the author name of each one. Without optimization, Django goes to the database to fetch each author individually:" %}

{% trans "Problematic code" %}

books = Book.objects.all()[:50]
# In template:
{% verbatim %}{% for book in books %}
  {{ book.author.name }}  {# query here! #}
{% endfor %}{% endverbatim %}

{% trans "Generated queries" %}

{% trans "query 1 - fetch books" %}
SELECT * FROM store_book LIMIT 50
{% trans "query 2, 3, 4 ... 51 - one author per book" %}
SELECT * FROM store_author WHERE id = 1
SELECT * FROM store_author WHERE id = 2
{% trans "... 48 more identical queries" %}
{% trans "Why is it dangerous?" %} {% trans "With 50 books it's 51 queries. With 1000 books it's 1001. The number grows linearly with the data - in production this can crash the database." %}

{% trans "How to identify" %}

  • {% trans "Django Debug Toolbar shows a high number of similar queries" %}
  • {% trans "Silk displays repeated queries with almost identical SQL" %}
  • connection.queries shows multiple queries WHERE id = N for the same table
{% trans "See N+1 FK in action" %} {% trans "See solution" %}

02{% trans "N+1 Problem - ManyToMany" %} {% trans "PROBLEM" %}

{% trans "The same N+1 occurs with ManyToMany relationships. The difference is that Django uses an intermediate table and accessing the relationship always triggers a new query, even if the data was already loaded before." %}

{% trans "How it happens" %}

{% trans "Problematic code" %}

books = Book.objects.all()[:50]
# In template:
{% verbatim %}{% for book in books %}
  {% for tag in book.tags.all %}  {# query here! #}
    {{ tag.name }}
  {% endfor %}
{% endfor %}{% endverbatim %}

{% trans "Generated queries" %}

{% trans "query 1 - fetch books" %}
SELECT * FROM store_book LIMIT 50
{% trans "queries 2 to 51 - tags of each book via intermediate table" %}
SELECT t.* FROM store_tag t
INNER JOIN store_book_tags bt ON t.id = bt.tag_id
WHERE bt.book_id = 1
{% trans "... 49 more queries" %}
{% trans "Difference from FK:" %} {% trans "With ForeignKey the JOIN can be done in the same query. With ManyToMany there are always at least 2 queries - but 2 is infinitely better than N+1." %}
{% trans "See N+1 M2M in action" %} {% trans "See solution" %}

03{% trans "Unnecessary SELECT *" %} {% trans "PROBLEM" %}

{% trans "By default, Django fetches all model fields (SELECT *). If your listing uses only 3 of the 10 available fields, you are transmitting unnecessary data between the database and the application - wasting I/O, memory and serialization time." %}

{% trans "How it happens" %}

{% trans "Problematic code" %}

books = Book.objects.all()
# loads: id, title, synopsis (TextField!),
# published_year, price, author_id
# even when using only title and price

{% trans "Generated query" %}

SELECT id, title, synopsis, published_year,
price, author_id
FROM store_book
{% trans "↑ synopsis can have kilobytes per row" %}
{% trans "Worst case:" %} {% trans "Fields like TextField, JSONField and BinaryField can contain megabytes of data. Loading them unnecessarily is one of the most costly mistakes in production." %}
{% trans "See SELECT * in action" %} {% trans "See solution" %}

04len() vs count() {% trans "PROBLEM" %}

{% trans "When you only need the number of records, using len() on a queryset forces Django to execute SELECT * and instantiate all objects in memory just to count them. The database already knows how to do this much more efficiently via COUNT(*)." %}

{% trans "Direct comparison" %}

{% trans "len() - loads everything" %}

books = Book.objects.all()
total = len(books)
# SELECT * FROM store_book
# instantiates N Book objects
# counts in Python memory

{% trans "count() - delegates to database" %}

total = Book.objects.count()
# SELECT COUNT(*) FROM store_book
# returns an integer
# no objects instantiated

{% trans "Other common cases of the same pattern" %}

{% trans "Check existence - wrong" %}

if len(Book.objects.filter(author=a)) > 0:
    ...
# or
if Book.objects.filter(author=a):
    ...

{% trans "Check existence - correct" %}

if Book.objects.filter(author=a).exists():
    ...
# SELECT 1 FROM store_book
# WHERE author_id = N LIMIT 1
{% trans "See len() in action" %} {% trans "See solution" %}

05{% trans "Missing pagination" %} {% trans "PROBLEM" %}

{% trans "Returning all records from a table in a single request is a classic mistake. Besides overloading the database with a SELECT without LIMIT, you load all objects into the application memory and send a huge HTTP response to the client." %}

{% trans "Impact in numbers" %}

  • {% trans "1,000 books × 500 bytes each = ~500 KB per request" %}
  • {% trans "10,000 books = ~5 MB - can timeout on client" %}
  • {% trans "100,000 books = ~50 MB - can crash the application" %}

{% trans "Without pagination" %}

books = Book.objects.all()
# SELECT * FROM store_book
# returns ALL records

{% trans "With pagination" %}

paginator = Paginator(qs, per_page=20)
page = paginator.get_page(request.GET.get("page"))
# SELECT * FROM store_book LIMIT 20 OFFSET 0

06select_related {% trans "SOLUTION" %}

select_related {% trans "select_related solves N+1 for ForeignKey and OneToOneField relationships. It instructs Django to perform a SQL JOIN, bringing the related object in the same query - without any extra queries." %}

{% trans "How it works internally" %}

Book.objects.select_related("author")[:50]
SELECT b.id, b.title, b.published_year, b.price,
a.id, a.name, a.country, a.born_year
FROM store_book b
INNER JOIN store_author a ON b.author_id = a.id
LIMIT 50

{% trans "Usage examples" %}

{% trans "Simple FK" %}

Book.objects.select_related("author")

{% trans "Chained FK (author → country)" %}

Book.objects.select_related("author__country")

{% trans "Multiple relationships" %}

Book.objects.select_related("author", "publisher")

{% trans "All FKs (careful with cartesian product)" %}

Book.objects.select_related()

{% trans "When to use" %}

  • ForeignKey {% trans "ForeignKey and OneToOneField relationships" %}
  • {% trans "When you always access the related object" %}
  • {% trans "When the relationship is to a single object (not a list)" %}
{% trans "Warning:" %} {% blocktrans %}Don't use select_related() without arguments on models with many FKs - it JOINs all of them, which can generate huge queries and slow things down. Prefer listing explicitly: select_related("author", "category").{% endblocktrans %}

07prefetch_related {% trans "SOLUTION" %}

prefetch_related {% trans "prefetch_related solves N+1 for ManyToManyField and reverse ForeignKey relationships (back relationships). Unlike select_related, it performs 2 separate queries and combines the results in Python - it's not possible to use JOIN for many-to-many relationships." %}

{% trans "How it works internally" %}

{% trans "query 1 - fetch books" %}
SELECT * FROM store_book LIMIT 50
{% trans "query 2 - fetch ALL tags for books at once" %}
SELECT t.*, bt.book_id FROM store_tag t
INNER JOIN store_book_tags bt ON t.id = bt.tag_id
WHERE bt.book_id IN (1, 2, 3, ..., 50)
↑ Django combines with books in memory

{% trans "Usage examples" %}

ManyToMany

Book.objects.prefetch_related("tags")

{% trans "Reverse FK (reviews of each book)" %}

Book.objects.prefetch_related("reviews")

{% trans "Combined with select_related" %}

Book.objects.select_related("author")\
            .prefetch_related("tags", "reviews")

{% trans "Prefetch with custom queryset" %}

from django.db.models import Prefetch
Book.objects.prefetch_related(
    Prefetch("reviews",
        queryset=Review.objects.filter(rating__gte=4),
        to_attr="top_reviews"
    )
)

{% trans "select_related vs prefetch_related - visual difference" %}

{% trans "Criteria" %} select_related prefetch_related
{% trans "Relationship type" %} FK, OneToOne M2M, reverse FK
{% trans "SQL mechanism" %} {% trans "JOIN in one query" %} {% trans "2 separate queries + Python" %}
{% trans "Number of queries" %} 1 {% trans "2 (independent of N)" %}
{% trans "Where data is combined" %} {% trans "Database" %} {% trans "Python (in memory)" %}
{% trans "Supports extra filters" %} {% trans "No" %} {% trans "Yes, via Prefetch()" %}
{% trans "Practical rule:" %} {% trans "If the relationship points to one object use select_related. If it points to many objects use prefetch_related." %}

08only() and defer() {% trans "SOLUTION" %}

{% trans "Both solve the unnecessary SELECT * problem, but from opposite perspectives: only() specifies which fields to include, defer() specifies which fields to exclude." %}

{% trans "only() - include only these" %}

Book.objects.only("title", "price")
# SELECT id, title, price FROM store_book
# id is always included automatically

{% trans "defer() - exclude only these" %}

Book.objects.defer("synopsis")
# SELECT id, title, price, published_year,
#        author_id FROM store_book
# synopsis is loaded only if accessed

{% trans "The lazy behavior of defer()" %}

{% trans "Fields deferred with defer() are not lost - they are loaded automatically if you access them, but this triggers an extra query per object. Use defer() only when you're sure the field will rarely be accessed." %}

book = Book.objects.defer("synopsis").get(id=1)
SELECT id, title, price FROM store_book WHERE id = 1
{% trans "access triggers new query!" %}
SELECT synopsis FROM store_book WHERE id = 1
{% trans "When to use each:" %} {% trans "Use only() when you know exactly which fields you need (listings, APIs). Use defer() when you want to exclude only a specific heavy field (like a TextField with HTML) but need the rest." %}

09values() and values_list() {% trans "SOLUTION" %}

{% trans "When you don't need model instances - only raw data - values() and values_list() return dictionaries and tuples directly, without the overhead of instantiating Python objects." %}

{% trans "values() - returns dictionaries" %}

Book.objects.values("id", "title", "price")
# [
#   {"id": 1, "title": "Django", "price": 49.90},
#   {"id": 2, "title": "Python", "price": 39.90},
# ]

{% trans "values_list() - returns tuples" %}

Book.objects.values_list("id", "title")
# [(1, "Django"), (2, "Python")]

# flat=True for a single column
Book.objects.values_list("title", flat=True)
# ["Django", "Python"]

{% trans "Ideal use cases" %}

  • {% trans "Populate an HTML select with id and name" %}
  • {% trans "Export data to CSV or spreadsheet" %}
  • {% trans "Pass data to a serializer (DRF, JSON)" %}
  • {% trans "Check if values exist" %}: values_list("email", flat=True)
{% trans "Limitation:" %} {% trans "Objects returned by values() and values_list() are simple dictionaries/tuples - you cannot call model methods on them (like get_absolute_url() or save()). If you need methods, use only() instead." %}

10annotate() {% trans "SOLUTION" %}

{% trans "annotate() adds calculated fields directly in the SQL query, using database aggregation functions. Instead of loading all related objects into Python to count, sum, or calculate averages, you delegate this operation to the database - which is much more efficient for this." %}

{% trans "Practical examples" %}

{% trans "Count reviews in Python - wrong" %}

for book in Book.objects.prefetch_related("reviews"):
    # still loads all Review objects
    total = book.reviews.count()  # extra query!

{% trans "Count reviews in database - correct" %}

from django.db.models import Count, Avg
books = Book.objects.annotate(
    num_reviews=Count("reviews"),
    avg_rating=Avg("reviews__rating"),
)
# access without extra query
book.num_reviews  # already in the object
book.avg_rating

{% trans "Available aggregation functions" %}

from django.db.models import Count, Sum, Avg, Max, Min, StdDev, Variance
# count
Book.objects.annotate(num_tags=Count("tags"))

# average price by author
Author.objects.annotate(avg_price=Avg("books__price"))

# most expensive book per author
Author.objects.annotate(max_price=Max("books__price"))

# total sales
Book.objects.annotate(total_sales=Sum("orders__quantity"))
Book.objects.annotate(num_reviews=Count("reviews"))[:5]
SELECT b.*, COUNT(r.id) AS num_reviews
FROM store_book b
LEFT OUTER JOIN store_review r ON b.id = r.book_id
GROUP BY b.id
LIMIT 5

11{% trans "Indexes" %} {% trans "SOLUTION" %}

{% trans "Indexes speed up search and filter queries. Without an index, the database does a Seq Scan - reads row by row until it finds what it needs. With an index, it does an Index Scan - goes directly to the data like a dictionary." %}

{% trans "How to define in Django" %}

{% trans "Simple index on one field" %}

class Book(models.Model):
    published_year = models.IntegerField(
        db_index=True  # creates index automatically
    )

{% trans "Composite index via Meta" %}

class Book(models.Model):
    class Meta:
        indexes = [
            models.Index(
                fields=["published_year", "price"],
                name="book_year_price_idx"
            ),
        ]

{% trans "How to check if the index is being used" %}

# via Django shell
from django.db import connection
qs = Book.objects.filter(published_year=2020)
print(qs.explain(verbose=True, analyze=True))

# via psql
docker compose exec db psql -U perfuser -d perfdb
EXPLAIN ANALYZE SELECT * FROM store_book WHERE published_year = 2020;
{% trans "When to create an index:" %} {% trans "on fields used in frequent filter(), order_by(), JOIN and DISTINCT. ForeignKeys already have automatic index in Django. Avoid indexing fields that change frequently - indexes have a write cost." %}

Django Silk

{% trans "Persistent profiling dashboard. Records all requests and database queries - useful for comparing performance over time and identifying slow endpoints." %} 

pip install django-silk

# settings.py
INSTALLED_APPS += ["silk"]
MIDDLEWARE = ["silk.middleware.SilkyMiddleware", ...MIDDLEWARE]
SILKY_PYTHON_PROFILER = True

# urls.py
path("silk/", include("silk.urls", namespace="silk"))

{% trans "Access at" %} /silk/.

connection.queries

{% trans "Python list with all queries from the current request. Only available with DEBUG=True. Useful in tests and diagnostic scripts." %} 

from django.db import connection

# see all queries
for q in connection.queries:
    print(q["sql"])
    print(q["time"])

# count
print(len(connection.queries))

QuerySet.explain()

{% trans "Returns the database execution plan for the query. Available since Django 2.1." %}

# simple plan
print(Book.objects.filter(published_year=2020).explain())

# detailed plan with real time (PostgreSQL)
print(Book.objects.filter(published_year=2020).explain(
    verbose=True,
    analyze=True,
    buffers=True,
))
{% trans "What to look for in EXPLAIN:" %} Seq Scan {% trans "Seq Scan indicates the database read the entire table (may need an index). Index Scan indicates the index was used. Hash Join and Nested Loop indicate how JOINs are being executed. cost= indicates the estimated cost (lower is better)." %}
